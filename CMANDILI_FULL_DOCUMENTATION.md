# PART A — `cmandili_mobile` (Customer App)

## A.1 App Overview

`cmandili_mobile` is the customer-facing Flutter mobile app of the Cmandili platform — a multi-service "super-app" targeting Tunisia (currency `DT`, locale `fr_TN`). Three companion apps share the same Supabase backend: this customer app, `cmandili_driver` (drivers), and `cmandili_partner` (restaurants/supermarkets).

**Services exposed to the user (from `lib/core/models/service_category.dart`):**

- **Food Delivery** — order from restaurants
- **Supermarket** — order groceries
- **Bill Payments** — request a driver to collect cash for utility bills (Ooredoo, Telecom Tunisia, Orange, STEG, SONEDE)
- **Courier** — peer-to-peer package delivery between two addresses

**Plus:** Happy Hour (time-limited discounts on food + groceries), Favorites, Order tracking with live driver location on Mapbox, push notifications, profile management, and an in-app support ticket system.

The app supports **English, Arabic, French** and light/dark theme. Auth is Supabase email/password and Google Sign-In. Apple Sign-In is implemented in the repository but not wired into the auth UI. Payment is Cash on Delivery only — `CashGateway` is the sole gateway implementation.

## A.2 Entry Point — `lib/main.dart`

**Initialization sequence (`main()` lines 20–45):**

1. `WidgetsFlutterBinding.ensureInitialized()`
2. `dotenv.load(fileName: '.env')` — loads `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `MAPBOX_PUBLIC_TOKEN`
3. `MapboxOptions.setAccessToken(...)` from dotenv
4. In parallel via `Future.wait`:
   - `Supabase.initialize(url: SupabaseConfig.url, anonKey: SupabaseConfig.anonKey)`
   - `Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform)` — project `cmandili-c1ade`, swallows duplicate-app errors
5. `runApp(ProviderScope(child: MyApp()))`
6. **Post first frame:** `PushService.instance.initialize()` (deferred so UI paints first)

`MyApp.build` watches `authStateProvider`, `localizationProvider`, `themeProvider`. Routing is `home`-based (no named routes):
- `authState.data == null` → `AuthScreen`
- `authState.data != null` → `HomeScreen`
- loading → centered spinner
- error → `AuthScreen`

`firebase_options.dart` carries the Android & iOS `FirebaseOptions` (project `cmandili-c1ade`, sender `785469526658`, iOS bundle `com.cmandili.mobile`).

## A.3 Core Layer — `lib/core/`

### `core/config/supabase_config.dart`
Reads `SUPABASE_URL` / `SUPABASE_ANON_KEY` from `--dart-define` first, falling back to dotenv.

### `core/config/supabase_service.dart`
Generic `SupabaseService` wrapper (provider `supabaseServiceProvider`) with helpers — `signOut`, generic `query(table, select, eq, limit, orderBy)`, realtime `subscribe(table, event, callback)`, `uploadFile(bucket, path, filePath)`, `getPublicUrl(bucket, path)`. Largely unused — every feature talks to `Supabase.instance.client` directly.

### `core/theme/app_colors.dart`
Static color palette: primary `#FF6B35`, secondary `#F7931E`, accent `#4ECDC4`, plus success/error/warning/info, and three gradients.

### `core/theme/app_theme.dart`
`AppTheme.lightTheme` + `darkTheme` (Material3) — AppBar, Card, ElevatedButton, TextButton, InputDecoration, Icon, Divider themes.

### `core/utils/currency_formatter.dart`
`CurrencyFormatter.formatPrice(double)` → `"1,234.00 DT"` using `NumberFormat('#,##0.00','fr_TN')`. `formatPriceCompact` → `"1.2K DT"` for ≥1000.

### `core/utils/location_service.dart`
Wraps `geolocator` + `geocoding`: `isLocationServiceEnabled`, `checkPermissions`, `getCurrentPosition`, `getAddressFromCoordinates(lat,lng)`, `getCoordinatesFromAddress(addr)`, `calculateDistance(...)` returning km.

### `core/providers/localization_provider.dart`
`StateNotifier<Locale>` persisting to `SharedPreferences` key `languageCode`. Default `Locale('en')`.

### `core/providers/theme_provider.dart`
`StateNotifier<ThemeMode>` persisting to `SharedPreferences` key `themeMode`. Methods `setTheme`, `toggleTheme`.

### `core/providers/service_provider.dart`
`SelectedServiceNotifier` — `StateNotifier<ServiceType>` defaulting to `ServiceType.foodDelivery`. Used by home segmented selector.

### `core/models/service_category.dart`
`ServiceType` enum (foodDelivery, supermarket, billPayments, courier). `ServiceCategory.categories` static list — only food and supermarket are populated.

### `core/payment/payment_gateway.dart`
Abstract `PaymentGateway` interface (`displayName`, `methodKey`, `processPayment`) + `PaymentResult` data class.

### `core/payment/cash_gateway.dart`
`CashGateway` — always succeeds, `methodKey='cash'`, `displayName='Cash on Delivery'`.

### `core/payment/payment_service.dart`
`PaymentService.gatewayFor(methodKey, ctx)` returns `CashGateway`. `processAndRecord` runs the gateway then INSERTs into Supabase `payments` table with columns `order_id, user_id, amount, method='cash', status='pending', gateway_ref, gateway_payload`.

### `core/widgets/app_map.dart`
`AppMap` is a Mapbox-backed map widget. Marker kinds: `delivery` (green), `pickup` (violet), `driver` (orange) — pin PNGs rasterized at runtime via Canvas. Controller exposes `animateToPoint`, `fitBounds`. Uses Mapbox `PointAnnotationManager` and `PolylineAnnotationManager`.

### `core/push/push_service.dart`
Singleton `PushService.instance`. `initialize()`:
1. Requests FCM permissions (alert/badge/sound)
2. Initializes `flutter_local_notifications`
3. Pre-creates Android channel `cmandili_orders`
4. Listens to `FirebaseMessaging.onMessage` (foreground → `_local.show(...)`) and `onBackgroundMessage`
5. Calls `_registerToken()` and on `onTokenRefresh`
6. On Supabase `AuthChangeEvent.signedIn`, re-registers

`_registerToken()` upserts into Supabase `device_tokens` with `user_id`, `token`, `platform=defaultTargetPlatform.name`, `onConflict: 'token'`.

# PART B — `cmandili_driver` (Driver App)

## B.1 App Overview

**Cmandili Driver** is a Flutter delivery-driver app. Target users: independent delivery drivers who pick up food, supermarket, and courier orders. Vehicle types: motorcycle, car, bicycle, scooter.

Capabilities:
- Authenticate (email/password or Google) into Supabase Auth
- Register a vehicle (mandatory gate before main app)
- Toggle online/offline
- Browse and accept available pending orders (realtime)
- Run an active delivery with live GPS streaming to Supabase (foreground service when backgrounded)
- Receive FCM push notifications for new orders
- View delivery history, period earnings, and payout/bank info
- Three-language UI (en, ar, fr) and light/dark theme

## B.2 Entry Point — `lib/main.dart`

Initialization sequence (`main.dart:24-59`):
1. `WidgetsFlutterBinding.ensureInitialized()`
2. `dotenv.load(fileName: '.env')` — loads `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `MAPBOX_PUBLIC_TOKEN`
3. `MapboxOptions.setAccessToken(...)`
4. Parallel `Future.wait`: `Supabase.initialize`, `Firebase.initializeApp`, `SharedPreferences.getInstance()`
5. Persists `supabase_url` / `supabase_anon_key` to `SharedPreferences` so the background isolate can read them
6. `runApp(ProviderScope(child: MyApp()))`
7. Post-frame: `PushService.instance.initialize()` and `BackgroundLocationService.initialize()`

`MyApp`: title `'Cmandili Driver'`, supportedLocales: `en, ar, fr`. Home keyed off `authStateProvider`:
- data + non-null user → `_PostAuthGate`
- data + null → `AuthScreen`
- loading → spinner; error → `AuthScreen`

`_PostAuthGate` checks `drivers.vehicle_type` for current user. If null/empty, blocks navigation with `VehicleInfoScreen(onSaved: _recheck)`. After save, re-runs `_checkVehicle()`. On success → `HomeScreen`.

### `core/services/background_location_service.dart`
Foreground service via `flutter_background_service` for cross-isolate GPS streaming.
- Constants: `_kDriverIdKey='bg_driver_id'`, `_kDeliveryIdKey='bg_delivery_id'`, `_kNotifChannelId='cmandili_driver_location'`, `_kNotifId=888`

- `_onStart` (background isolate): reads IDs from prefs, re-initializes Supabase. Position stream + 10-second `Timer.periodic` heartbeat:
  - Updates Android foreground notification text
  - `drivers.update({current_lat, current_lng, last_location_update})` by `id`
  - If deliveryId set: `deliveries.update({current_lat, current_lng, updated_at})` by `id`

# PART D — Supabase Backend

## D.1 Edge Function — `supabase/functions/push-on-order-status/index.ts`

Deno edge function. Central FCM fan-out for the platform. Invoked exclusively by Postgres triggers via `pg_net.http_post` with bearer token from database GUCs.

### Invocation modes

Dispatches on `event` field in JSON body:

**Mode A — status change (default, `event` absent or `"status"`):** triggered by `notify_fcm_on_order_status` (UPDATE OF status) and `notify_fcm_on_new_order` (INSERT). Body: `{ order_id, status }`. Fans out to up to three audiences:
- **Customer** — `orders.user_id`
- **Partner** — resolved by querying `partners` where `partner_type IN ('restaurant', 'supermarket')` and `entity_id = orders.restaurant_id` (or `supermarket_id`). `entity_id` is TEXT joined to UUIDs as text.
- **Assigned driver** — if `orders.driver_id` non-null, fetch `drivers.user_id`

**Mode B — driver fan-out (`event = "driver_fanout"`):** triggered by `notify_fcm_fanout_ready_order` when status transitions into `'ready'`. Body: `{ event: "driver_fanout", order_id, status: "ready" }`. The function:
1. Reads order for `restaurant_id`, `supermarket_id`, `pickup_address`
2. Resolves pickup lat/lng from `restaurants` or `supermarkets` (or `pickup_address.lat/lng` for courier)
3. Calls `nearby_online_drivers(p_lat, p_lng, p_radius_km)` RPC. Default 7 km, configurable via `DRIVER_FANOUT_RADIUS_KM`
4. Pushes "🔔 New delivery nearby" to every returned driver

### Push copy tables

Three role-keyed dictionaries (`CUSTOMER_COPY`, `PARTNER_COPY`, `DRIVER_COPY`) map status string → `{title, body}`.

- Customer: `confirmed`, `preparing`, `ready`, `pickedUp`, `onTheWay`, `delivered`, `cancelled`
- Partner: `pending`, `confirmed`, `ready`, `pickedUp`, `delivered`, `cancelled`
- Driver: `pickedUp`, `onTheWay`, `delivered`, `cancelled`

Unknown statuses fall back to `"Order Update" / "Status: <status>"`.

### FCM wiring

`getAccessToken()` builds JWT (`alg: RS256`) signed with service account `private_key` via `crypto.subtle.importKey('pkcs8', …)` + `RSASSA-PKCS1-v1_5 / SHA-256`. Exchanged at `https://oauth2.googleapis.com/token` with grant `urn:ietf:params:oauth:grant-type:jwt-bearer`, scope `https://www.googleapis.com/auth/firebase.messaging`.

`sendFcm()` POSTs to `https://fcm.googleapis.com/v1/projects/{project_id}/messages:send` with envelope `{ message: { token, notification, data, android: {priority: 'high'}, apns: {headers: {'apns-priority': '10'}} } }`. Failures absorbed via `Promise.allSettled`.

### Token resolution

`tokensForUser(supabase, userId)` selects from `device_tokens` for user. `pushToUsers` deduplicates flattened token set, sends in parallel.

### Required env vars

- `SUPABASE_URL` — auto-injected
- `SERVICE_ROLE_KEY` — service-role JWT. **Cannot be named `SUPABASE_SERVICE_KEY`** — Supabase reserves `SUPABASE_*` prefix
- `FCM_SERVICE_ACCOUNT_JSON` — base64-encoded Firebase service account JSON
- `DRIVER_FANOUT_RADIUS_KM` — optional, default 7

### Side effects

- Hits Firebase OAuth (1 token exchange per invocation, not cached)
- Hits FCM v1 send endpoint once per device token
- Reads: `orders`, `restaurants`, `supermarkets`, `partners`, `drivers`, `device_tokens`
- Calls `nearby_online_drivers` RPC in fan-out mode
- Returns `{ mode, customer?, partner?, driver?, sent? }`

## D.5 Push Notification Flow (end-to-end)

From a partner tapping "Mark Ready" through driver phones vibrating:

1. **Client write.** Partner app calls `UPDATE orders SET status = 'ready' WHERE id = …`. RLS allows because partner's row in `partners` matches order's `restaurant_id`/`supermarket_id`.

2. **`BEFORE UPDATE` trigger** `order_status_timestamps` runs → stamps `ready_at = now()`.

3. **Three `AFTER UPDATE OF status` triggers** fire:
   - `on_order_status_changed` → `handle_order_status_change()` inserts row into `notifications` (in-app feed)
   - `on_order_status_push` → `notify_fcm_on_order_status()` POSTs `{order_id, status}` to edge function via `pg_net`
   - `on_order_ready_driver_fanout` → `notify_fcm_fanout_ready_order()` (only entering `ready`) POSTs `{event: 'driver_fanout', order_id, status: 'ready'}`

4. **Edge function (Mode A)**: reads order, resolves customer/partner/driver, looks up `device_tokens`, sends FCM v1 messages with role-specific copy.

5. **Edge function (Mode B)**: resolves pickup coords (restaurants/supermarkets/`pickup_address.lat/lng`), calls `nearby_online_drivers(lat, lng, 7)`, pushes "🔔 New delivery nearby" to every returned driver.

6. **New-order push** (added by 20260507): on customer order creation, `on_order_insert_push` → `notify_fcm_on_new_order()` POSTs `{order_id, status: 'pending'}`. Mode A runs and partner receives "🔔 New Order".

7. **Customer-side downstream flows** (`confirmed → preparing → pickedUp → onTheWay → delivered`): identical Mode A path.

8. **Token registration:** Flutter apps write to `device_tokens` on app start. Tokens are user-scoped.

9. **Realtime channel** runs in parallel — `drivers`, `orders`, `deliveries` are in `supabase_realtime` publication, so customer's tracking screen subscribes to driver location via `deliveries.current_lat/lng` updates.

## E.3 Order status state machine (canonical)

```
pending → confirmed → preparing → ready → pickedUp → onTheWay → delivered
   ↓          ↓           ↓          ↓
  cancelled (any pre-pickup state)
```

| Actor | Transitions written |
|---|---|
| Customer | creates `pending` (insert), can confirm to `confirmed`, can cancel to `cancelled`, can confirm receipt → `delivered` (courier) |
| Partner | `pending → confirmed → preparing → ready` (and any → `cancelled` pre-pickup) |
| Driver | `ready → pickedUp → onTheWay → delivered` (also accepts by writing `driver_id`, no status change) |
| System | `BEFORE UPDATE` trigger stamps `confirmed_at`/`ready_at` |
